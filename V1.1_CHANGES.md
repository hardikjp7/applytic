## Deploy Order

### Step 1 — Run OIDC setup (one-time, manual)
```bash
cd scripts
chmod +x setup_oidc_v11.sh
./setup_oidc_v11.sh
```
Copy the output Role ARN.

### Step 2 — Update GitHub Secrets
- ADD: `AWS_DEPLOY_ROLE_ARN` = the ARN from step 1
- DELETE: `AWS_ACCESS_KEY_ID`
- DELETE: `AWS_SECRET_ACCESS_KEY`

### Step 3 — Generate lockfiles (needed for npm caching)
```bash
cd cdk && npm install   # generates cdk/package-lock.json
cd ../frontend && npm install  # generates frontend/package-lock.json
git add cdk/package-lock.json frontend/package-lock.json
git commit -m "chore: add package-lock files for CI caching"
```

### Step 4 — Deploy CDK stack (picks up TTL + CORS + alarms)
```bash
cd cdk && cdk deploy
```
After deploy, confirm the SNS subscription email (check hardikjparmar7@gmail.com).

### Step 5 — Commit and push to develop, then PR to main
```bash
git checkout develop
# copy files to correct locations
git add .
git commit -m "feat/v11: OIDC migration, CORS tighten, DynamoDB TTL, CloudWatch alarms, Dependabot, CodeQL"
git push origin develop
# open PR: develop → main
```

## What Each Change Does

**OIDC** — GitHub Actions no longer uses long-lived AWS access keys stored as secrets.
Instead, it gets a short-lived token per job via AWS IAM's OIDC federation. Keys that
can't be leaked can't be compromised.

**npm/pip caching** — `setup-node` and `setup-python` cache node_modules and pip wheels
between runs. Saves ~60-90 seconds per job on warm runs.

**Concurrency controls** — if you push twice in 30 seconds, the first run is cancelled
automatically. No more two deployments racing each other.

**CORS tightened** — API Gateway now only allows requests from your CloudFront domain
and GitHub Pages origin. `*` wildcard removed.

**Dependabot** — automated weekly PRs for npm (cdk, frontend), pip (lambdas), and
GitHub Actions versions. Grouped by ecosystem to reduce noise.

**CodeQL** — static security analysis on every push and PR for both Python and
TypeScript. Runs on a schedule (Monday) as well.

**DynamoDB TTL** — the `ttl` attribute was already being written by the rate limiter
in insights Lambda (set to 2 days ahead). CDK now enables TTL on the table so DynamoDB
automatically deletes expired rate limit records. No cost, no Lambda needed.

**CloudWatch alarms** — 5 alarms:
- Applications Lambda error count > 5 in 5 min → SNS email
- Insights Lambda error count > 5 in 5 min → SNS email
- Digest Lambda any error → SNS email (scheduled jobs failing silently is bad)
- Applications Lambda p99 > 10s → SNS email
- Insights Lambda p99 > 30s → SNS email (Bedrock calls are slow by nature)

All alarms notify via SNS → email to hardikjparmar7@gmail.com.
